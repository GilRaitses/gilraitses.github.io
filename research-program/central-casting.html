Central Casting

A recent, still-evolving method for keeping multi-repository agent work coherent.

The research collected in this gallery moves across several repositories at once. Over the last few months I started formalizing how that work stays coherent between sessions, so a new session can resume from recorded state. This page describes the current approach at the level of working structure. It is recent and still being tested, while parts of it remain as much intention as settled habit.

Timeline, honestly

The pipelines and analysis tools came first over several years of ordinary research work. The coordination layer described below is the newer piece and still early. The checkpoint method, the step logs and the separation of roles were written in spring 2026. Before that period, continuity across sessions was mostly an ad hoc recovery habit. So this is an early practice built on older work, young enough that its habits are still forming.

Checkpoints as memory

The current unit of memory is a checkpoint, a recorded state change in what a part of the work knows, can do, is blocked by or is allowed to write. A checkpoint marks a real transition in state, at a coarser grain than a chat turn and a finer grain than task completion. A session reads the most recent checkpoints from disk and resumes work from there. The discipline is young, so it holds when it is followed and loosens when attention slips.

Agent homes as role contracts

Each part of the work has a defined home with explicit inputs, outputs and an escalation rule, separating orchestration, execution and research. Handoffs are written down so one step can take over from another without losing context. This structure is newer than the research it coordinates and is still settling.

A deliberate local and cloud boundary

The memory and audit layer stays under local control by design, so the connective record does not wander into a remote workspace by accident. Production work such as formatting, packaging and cross-repository assembly can be delegated when the load justifies it. Keeping that boundary explicit is a durable design choice for the practice. The connective tissue stays under local control, while heavier production steps can run elsewhere.

Dispatching a waveset

The newest layer lets one orchestrator run a whole campaign without doing the work itself. The orchestrator holds the map and writes a charter for the campaign, a waveset, then dispatches that charter to a fresh orchestrator, a background session or a separate thread. That dispatched orchestrator runs the campaign as waves of parallel subagents, where a subagent is the same kind of worker a part of the work already spawns. Each subagent owns one bounded task and one separate set of files. The dispatched orchestrator reports each wave back and answers to the orchestrator that sent it, not to the operator, so escalation stays inside the method as a safety property. This reuses the existing roles and does not invent a new tier of machinery for the same job. It is the most recent part of the practice and the least settled, so the honest claim stays modest.

What is still hard

Context still drifts across long sessions, while a change in one repository reaches another only when it is carried there by hand. The checkpoint discipline reduces these failures while leaving some in place, while it still leans on care ahead of automation. Because the method is still recent, the honest claim about its reliability stays deliberately modest. It helps in daily use, while the practice is still being proven under real campaigns.
